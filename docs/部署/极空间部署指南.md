# 极空间部署操作文档

> 适用场景：在极空间 NAS 上通过 Docker / Compose 部署 Classifier 第一阶段版本。

## 1. 部署前准备

需要准备以下信息：

| 项目 | 说明 |
|---|---|
| 极空间 IP | 例如 `192.168.1.20` |
| 源目录 | 待扫描目录，建议只读挂载 |
| 目标目录 | 分类完成后的移动目标目录 |
| Docker 环境 | 极空间已启用 Docker |

建议先在极空间文件管理器中确认真实挂载路径。

常见路径形态如下：

```text
/tmp/zfsv3/sdX/data/<share_name>/...
```

例如：

```text
/tmp/zfsv3/sda1/data/media/incoming
/tmp/zfsv3/sda1/data/media/processed
```

## 2. 为什么默认使用 root 运行

极空间的共享目录 ACL 和 Docker bind mount 组合下，很多容器在非 root 用户运行时会遇到：

- 能读取但不能删除
- 能进入目录但不能重命名
- 能写子文件但不能移动整个目录

因此当前默认方案使用：

```yaml
user: "0:0"
```

这是为了优先保证以下能力可用：

- 扫描目录
- 访问文件夹内容
- 移动目录
- 删除或回滚目录

当前不默认开启：

- `privileged: true`

因为第一阶段功能不需要额外设备权限或宿主机控制能力。

## 3. 环境变量配置

先复制配置模板：

```bash
cp .env.example .env
```

然后编辑 `.env`：

```env
APP_PORT=8080
SOURCE_DIR=/tmp/zfsv3/sda1/data/media/incoming
TARGET_DIR=/tmp/zfsv3/sda1/data/media/processed
MAX_CONCURRENCY=3
TZ=Asia/Shanghai
CPU_LIMIT=2.0
MEMORY_LIMIT=1G
```

推荐原则：

- `SOURCE_DIR` 指向待扫描目录
- `TARGET_DIR` 指向分类后落盘目录
- 两个目录不要互相嵌套

## 3.1 想在目录选择器里看到 NAS 全部目录，必须这样配

很多用户以为只填 `SOURCE_DIR` 就能浏览全盘，这是不对的。  
目录选择器能看到哪些路径，取决于 **容器挂载了哪些宿主机目录**。

当前项目使用：

- `${NAS_ROOT}:${NAS_ROOT}:rw`

因此要让前端目录选择器可浏览 NAS 全部共享目录，`NAS_ROOT` 必须指向极空间存储根路径，例如：

```env
NAS_ROOT=/tmp/zfsv3
SOURCE_DIR=/tmp/zfsv3/sda1/data
TARGET_DIR=/tmp/zfsv3/sda1/data/processed
```

说明：

- `NAS_ROOT=/tmp/zfsv3`：容器可访问该路径下所有盘与共享目录
- `SOURCE_DIR`：仅作为初始扫描目录，不限制目录选择器可见范围
- `TARGET_DIR`：仅作为初始目标目录，不限制目录选择器可见范围

如果你把 `NAS_ROOT` 设成某个子目录（例如 `/tmp/zfsv3/sda1/data/media`），目录选择器就只能看到这个子目录内的内容。

## 4. 当前挂载方式说明

当前 Compose 的目录映射为：

| 宿主机 | 容器内 | 模式 |
|---|---|---|
| `${NAS_ROOT}` | `${NAS_ROOT}` | 读写（全盘） |
| named volume | `/config` | 读写 |
| tmpfs | `/tmp/work` | 临时工作目录 |

`NAS_ROOT` 设置为极空间存储根路径（如 `/tmp/zfsv3`），容器内可访问该路径下所有共享目录，无需为每个目录单独挂载。`SOURCE_DIR` 和 `TARGET_DIR` 仍通过环境变量传入，作为初始配置使用，但必须位于 `NAS_ROOT` 路径下。

### 4.1 快速自检：容器是否真的拿到 NAS 根路径

启动后执行：

```bash
docker compose --env-file .env exec classifier sh -lc "ls -la ${NAS_ROOT}"
```

如果能看到盘符目录（如 `sda1`、`sdb1`）或你实际共享目录，说明挂载正确。  
如果报 `No such file or directory` 或权限错误，优先检查：

1. `.env` 中 `NAS_ROOT` 是否写成真实路径
2. `docker compose` 启动时是否带了 `--env-file .env`
3. 是否保留 `user: "0:0"`

后端读取的目录来自环境变量：

- `SOURCE_DIR`
- `TARGET_DIR`
- `CONFIG_DIR=/config`

数据库文件默认会写到：

```text
/config/classifier.db
```

## 5. 构建与启动

在仓库根目录执行：

```bash
docker compose --env-file .env build
docker compose --env-file .env up -d
```

查看日志：

```bash
docker compose --env-file .env logs -f
```

访问地址：

```text
http://<极空间IP>:8080
```

## 6. 首次启动后的检查项

建议按顺序检查：

### 6.1 健康检查

```bash
curl http://<极空间IP>:8080/health
```

期望返回：

```json
{"status":"ok"}
```

### 6.2 配置页

打开网页后进入 `Settings` 页面，确认：

- `source_dir` 正确
- `target_dir` 正确

### 6.3 扫描权限

点击扫描后，应能看到 SSE 推送的扫描进度，并在 Folder 列表中出现扫描结果。

如果完全扫不到内容，优先检查：

- `SOURCE_DIR` 是否写成极空间真实路径
- 该路径是否真的是目录
- 容器是否对该目录有读取权限

### 6.4 移动权限

执行移动或回滚时，如果出现失败，优先检查：

- `TARGET_DIR` 是否可写
- 源/目标目录是否在极空间 ACL 下允许当前容器用户修改
- 是否误把目标目录挂成了只读

## 7. 常见问题排查

### 7.1 报错：没有权限删除或移动

优先确认 `docker-compose.yml` 中仍然是：

```yaml
user: "0:0"
```

如果你改成了普通用户，先改回 root 再验证。

### 7.2 报错：`invalid spec: ::ro`

这是因为没有正确加载 `.env`，导致 `SOURCE_DIR` 或 `TARGET_DIR` 为空。

请使用：

```bash
docker compose --env-file .env build
docker compose --env-file .env up -d
```

不要省略 `--env-file .env`。

### 7.3 报错：Docker daemon 未启动

如果出现：

```text
Cannot connect to the Docker daemon
```

说明当前设备上的 Docker 服务未启动，需要先在极空间或当前终端环境中启动 Docker。

### 7.4 扫描正常，但移动失败

这通常不是程序逻辑问题，而是目录权限问题。建议：

1. 保持 `user: "0:0"`
2. 确认 `TARGET_DIR` 是共享目录真实路径
3. 手工在宿主机上确认该目录允许写入和重命名

## 8. 升级方式

更新代码后执行：

```bash
git pull
docker compose --env-file .env build
docker compose --env-file .env up -d
```

配置和数据库仍保留在 `/config` 对应的 volume 中。

## 9. 安全建议

当前为了极空间兼容性，容器默认以 root 运行。这是功能优先的折中方案。

如果后续你已经确认：

- 共享目录 ACL 已正确配置
- 普通 UID/GID 能完成扫描、移动、删除、回滚

可以再尝试将 `user: "0:0"` 替换为实际用户映射。但在没有验证前，不建议修改。

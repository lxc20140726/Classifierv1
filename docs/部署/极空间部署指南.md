# 极空间部署操作文档

> 适用场景：在极空间 NAS 上通过 Docker / Compose 部署 Classifier 第一阶段版本。

## 推荐部署方式：本地打包镜像 + 极空间 Compose 运行

推荐不要在极空间上构建源码，而是在本地电脑完成镜像构建与导出，然后把镜像包、`.env`、`docker-compose.image.yml` 上传到极空间，再用 Compose 启动。

整体流程：

```text
本地电脑构建镜像 -> 导出 classifier-*.tar -> 上传极空间 -> docker load -> docker compose up -d
```

这种方式的优点：

- 极空间不需要安装 Node.js、Go、git 等构建环境
- NAS CPU 压力更小，部署更稳定
- 极空间上只保留运行配置和镜像包，后续升级也更清晰

如果使用极空间 Docker 图形界面的“创建项目 / 添加方式：导入”，推荐直接使用仓库根目录的：

```text
docker-compose.zspace.yml
```

这个文件是极空间 UI 专用模板：

- 不依赖 `.env`
- 不包含 `build:`
- 直接使用已导入或可拉取的镜像
- 配置和数据库落在 NAS 可见目录 `/tmp/zfsv3/sda1/data/docker/classifier/config`

### 0.0 使用极空间图形界面创建 Compose 项目

在极空间 Docker 的“创建项目”窗口中：

| 字段 | 建议填写 |
|---|---|
| 项目名称 | `classifier` |
| 存储位置 | 例如 `/tmp/zfsv3/sda1/data/docker/classifier` |
| 添加方式 | 选择“导入” |
| Compose 内容 | 粘贴 `docker-compose.zspace.yml` 内容 |

粘贴前请先按你的实际目录修改这几项：

```yaml
SOURCE_DIR: /tmp/zfsv3/sda1/data/media/incoming
TARGET_DIR: /tmp/zfsv3/sda1/data/media/processed
DELETE_STAGING_DIR: /tmp/zfsv3/sda1/data/media/classifier-delete-staging
```

如果你的极空间存储根路径不是 `/tmp/zfsv3`，也要同步修改：

```yaml
volumes:
  - /tmp/zfsv3:/tmp/zfsv3:rw
```

图形界面 Compose 模板如下：

```yaml
services:
  classifier:
    image: classifier:local
    container_name: classifier
    user: "0:0"
    ports:
      - "8080:8080"
    environment:
      CONFIG_DIR: /tmp/zfsv3/sata12/public/Tran/classifier-config
      SOURCE_DIR: /tmp/zfsv3/sata12/public/Tran/待定
      TARGET_DIR: /tmp/zfsv3/sata12/public/Tran/处理完成
      DELETE_STAGING_DIR: /tmp/zfsv3/sata12/public/Tran/classifier-delete-staging
      MAX_CONCURRENCY: "3"
      TZ: Asia/Shanghai
    volumes:
      - /tmp/zfsv3/sata12/public/Tran:/tmp/zfsv3/sata12/public/Tran:rw
    tmpfs:
      - /tmp/work
    cpus: 2.0
    mem_limit: 1g
    restart: unless-stopped
```

重要说明：Compose 只能负责创建和运行容器，不能直接把 `classifier-local-linux-amd64.tar` 作为镜像源。使用图形界面前必须先满足以下任一条件：

1. 已经把 tar 镜像导入极空间 Docker 镜像列表，镜像名为 `classifier:local`
2. 已经把镜像推送到 Docker Hub / 私有镜像仓库，并把 `image:` 改成仓库地址

### 0.1 确认极空间 CPU 架构

先在极空间 SSH 里执行：

```bash
uname -m
```

常见对应关系：

| `uname -m` 输出 | 本地打包平台参数 |
|---|---|
| `x86_64` | `linux/amd64` |
| `aarch64` | `linux/arm64` |

后续命令里的 `DOCKER_PLATFORM` 要按这个结果设置。

### 0.2 在本地电脑打包镜像

Windows PowerShell：

```powershell
$env:CLASSIFIER_IMAGE="classifier"
$env:CLASSIFIER_TAG="local"
$env:DOCKER_PLATFORM="linux/amd64"
.\package-image.ps1
```

macOS / Linux：

```bash
CLASSIFIER_IMAGE=classifier \
CLASSIFIER_TAG=local \
DOCKER_PLATFORM=linux/amd64 \
sh ./script/package-image.sh
```

如果极空间是 ARM64，把 `linux/amd64` 改成：

```text
linux/arm64
```

打包完成后会生成类似文件：

```text
output/classifier-local-linux-amd64.tar
```

### 0.3 准备极空间运行目录

在极空间上创建一个部署目录，例如：

```bash
mkdir -p /tmp/zfsv3/sda1/data/docker/classifier
```

上传以下文件到这个目录：

```text
classifier-local-linux-amd64.tar
docker-compose.image.yml
.env
```

可以用 `scp`，也可以直接通过极空间文件管理器上传。

示例：

```bash
scp output/classifier-local-linux-amd64.tar docker-compose.image.yml .env root@<极空间IP>:/tmp/zfsv3/sda1/data/docker/classifier/
```

### 0.4 编辑极空间上的 `.env`

示例：

```env
APP_PORT=8080
CLASSIFIER_IMAGE=classifier
CLASSIFIER_TAG=local

NAS_ROOT=/tmp/zfsv3
SOURCE_DIR=/tmp/zfsv3/sda1/data/media/incoming
TARGET_DIR=/tmp/zfsv3/sda1/data/media/processed
DELETE_STAGING_DIR=/tmp/zfsv3/sda1/data/media/classifier-delete-staging

MAX_CONCURRENCY=3
TZ=Asia/Shanghai
CPU_LIMIT=2.0
MEMORY_LIMIT=1G
```

注意：

- `CLASSIFIER_IMAGE` 和 `CLASSIFIER_TAG` 必须和本地打包时一致
- `NAS_ROOT` 是容器可浏览的极空间存储根路径
- `SOURCE_DIR`、`TARGET_DIR`、`DELETE_STAGING_DIR` 建议都位于 `NAS_ROOT` 下面

### 0.5 在极空间加载镜像并启动

进入部署目录：

```bash
cd /tmp/zfsv3/sda1/data/docker/classifier
```

加载镜像：

```bash
docker load -i classifier-local-linux-amd64.tar
```

启动：

```bash
docker compose -f docker-compose.image.yml --env-file .env up -d
```

查看日志：

```bash
docker compose -f docker-compose.image.yml --env-file .env logs -f
```

访问地址：

```text
http://<极空间IP>:8080
```

### 0.6 后续升级

本地重新打包一个新 tag，例如：

```powershell
$env:CLASSIFIER_IMAGE="classifier"
$env:CLASSIFIER_TAG="2026-04-19"
$env:DOCKER_PLATFORM="linux/amd64"
.\script\package-image.ps1
```

上传新 tar 到极空间后：

```bash
docker load -i classifier-2026-04-19-linux-amd64.tar
```

修改 `.env`：

```env
CLASSIFIER_TAG=2026-04-19
```

重启：

```bash
docker compose -f docker-compose.image.yml --env-file .env up -d
```

配置和数据库保存在 `classifier_config` volume 中，不会因为换镜像而丢失。

## 1. 部署前准备

需要准备以下信息：

| 项目 | 说明 |
|---|---|
| 极空间 IP | 例如 `192.168.1.20` |
| 源目录 | 待扫描目录，建议只读挂载 |
| 目标目录 | 分类完成后的移动目标目录 |
| Docker 环境 | 极空间已启用 Docker |

建议先在极空间文件管理器中确认真实挂载路径。

常见路径形态如下：

```text
/tmp/zfsv3/sdX/data/<share_name>/...
```

例如：

```text
/tmp/zfsv3/sda1/data/media/incoming
/tmp/zfsv3/sda1/data/media/processed
```

## 2. 为什么默认使用 root 运行

极空间的共享目录 ACL 和 Docker bind mount 组合下，很多容器在非 root 用户运行时会遇到：

- 能读取但不能删除
- 能进入目录但不能重命名
- 能写子文件但不能移动整个目录

因此当前默认方案使用：

```yaml
user: "0:0"
```

这是为了优先保证以下能力可用：

- 扫描目录
- 访问文件夹内容
- 移动目录
- 删除或回滚目录

当前不默认开启：

- `privileged: true`

因为第一阶段功能不需要额外设备权限或宿主机控制能力。

## 3. 环境变量配置

先复制配置模板：

```bash
cp .env.example .env
```

然后编辑 `.env`：

```env
APP_PORT=8080
SOURCE_DIR=/tmp/zfsv3/sda1/data/media/incoming
TARGET_DIR=/tmp/zfsv3/sda1/data/media/processed
MAX_CONCURRENCY=3
TZ=Asia/Shanghai
CPU_LIMIT=2.0
MEMORY_LIMIT=1G
```

推荐原则：

- `SOURCE_DIR` 指向待扫描目录
- `TARGET_DIR` 指向分类后落盘目录
- 两个目录不要互相嵌套

## 3.1 想在目录选择器里看到 NAS 全部目录，必须这样配

很多用户以为只填 `SOURCE_DIR` 就能浏览全盘，这是不对的。  
目录选择器能看到哪些路径，取决于 **容器挂载了哪些宿主机目录**。

当前项目使用：

- `${NAS_ROOT}:${NAS_ROOT}:rw`

因此要让前端目录选择器可浏览 NAS 全部共享目录，`NAS_ROOT` 必须指向极空间存储根路径，例如：

```env
NAS_ROOT=/tmp/zfsv3
SOURCE_DIR=/tmp/zfsv3/sda1/data
TARGET_DIR=/tmp/zfsv3/sda1/data/processed
```

说明：

- `NAS_ROOT=/tmp/zfsv3`：容器可访问该路径下所有盘与共享目录
- `SOURCE_DIR`：仅作为初始扫描目录，不限制目录选择器可见范围
- `TARGET_DIR`：仅作为初始目标目录，不限制目录选择器可见范围

如果你把 `NAS_ROOT` 设成某个子目录（例如 `/tmp/zfsv3/sda1/data/media`），目录选择器就只能看到这个子目录内的内容。

## 4. 当前挂载方式说明

当前 Compose 的目录映射为：

| 宿主机 | 容器内 | 模式 |
|---|---|---|
| `${NAS_ROOT}` | `${NAS_ROOT}` | 读写（全盘） |
| named volume | `/config` | 读写 |
| tmpfs | `/tmp/work` | 临时工作目录 |

`NAS_ROOT` 设置为极空间存储根路径（如 `/tmp/zfsv3`），容器内可访问该路径下所有共享目录，无需为每个目录单独挂载。`SOURCE_DIR` 和 `TARGET_DIR` 仍通过环境变量传入，作为初始配置使用，但必须位于 `NAS_ROOT` 路径下。

### 4.1 快速自检：容器是否真的拿到 NAS 根路径

启动后执行：

```bash
docker compose --env-file .env exec classifier sh -lc "ls -la ${NAS_ROOT}"
```

如果能看到盘符目录（如 `sda1`、`sdb1`）或你实际共享目录，说明挂载正确。  
如果报 `No such file or directory` 或权限错误，优先检查：

1. `.env` 中 `NAS_ROOT` 是否写成真实路径
2. `docker compose` 启动时是否带了 `--env-file .env`
3. 是否保留 `user: "0:0"`

后端读取的目录来自环境变量：

- `SOURCE_DIR`
- `TARGET_DIR`
- `CONFIG_DIR=/config`

数据库文件默认会写到：

```text
/config/classifier.db
```

## 5. 构建与启动

在仓库根目录执行：

```bash
docker compose --env-file .env build
docker compose --env-file .env up -d
```

查看日志：

```bash
docker compose --env-file .env logs -f
```

访问地址：

```text
http://<极空间IP>:8080
```

## 6. 首次启动后的检查项

建议按顺序检查：

### 6.1 健康检查

```bash
curl http://<极空间IP>:8080/health
```

期望返回：

```json
{"status":"ok"}
```

### 6.2 配置页

打开网页后进入 `Settings` 页面，确认：

- `source_dir` 正确
- `target_dir` 正确

### 6.3 扫描权限

点击扫描后，应能看到 SSE 推送的扫描进度，并在 Folder 列表中出现扫描结果。

如果完全扫不到内容，优先检查：

- `SOURCE_DIR` 是否写成极空间真实路径
- 该路径是否真的是目录
- 容器是否对该目录有读取权限

### 6.4 移动权限

执行移动或回滚时，如果出现失败，优先检查：

- `TARGET_DIR` 是否可写
- 源/目标目录是否在极空间 ACL 下允许当前容器用户修改
- 是否误把目标目录挂成了只读

## 7. 常见问题排查

### 7.1 报错：没有权限删除或移动

优先确认 `docker-compose.yml` 中仍然是：

```yaml
user: "0:0"
```

如果你改成了普通用户，先改回 root 再验证。

### 7.2 报错：`invalid spec: ::ro`

这是因为没有正确加载 `.env`，导致 `SOURCE_DIR` 或 `TARGET_DIR` 为空。

请使用：

```bash
docker compose --env-file .env build
docker compose --env-file .env up -d
```

不要省略 `--env-file .env`。

### 7.3 报错：Docker daemon 未启动

如果出现：

```text
Cannot connect to the Docker daemon
```

说明当前设备上的 Docker 服务未启动，需要先在极空间或当前终端环境中启动 Docker。

### 7.4 扫描正常，但移动失败

这通常不是程序逻辑问题，而是目录权限问题。建议：

1. 保持 `user: "0:0"`
2. 确认 `TARGET_DIR` 是共享目录真实路径
3. 手工在宿主机上确认该目录允许写入和重命名

## 8. 升级方式

更新代码后执行：

```bash
git pull
docker compose --env-file .env build
docker compose --env-file .env up -d
```

配置和数据库仍保留在 `/config` 对应的 volume 中。

## 9. 安全建议

当前为了极空间兼容性，容器默认以 root 运行。这是功能优先的折中方案。

如果后续你已经确认：

- 共享目录 ACL 已正确配置
- 普通 UID/GID 能完成扫描、移动、删除、回滚

可以再尝试将 `user: "0:0"` 替换为实际用户映射。但在没有验证前，不建议修改。

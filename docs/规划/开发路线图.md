## 已完成（Phase 2.5）— 引擎兼容性修复（Phase 3 前置）

> Phase 2 实现的引擎与 `docs/功能/节点系统设计.md` 存在两处**阻断性差距**（不修复则分类管道节点无法运行）和若干**非阻断性 DB 缺口**（可用追加迁移补齐）。Phase 3 开工前必须完成本节全部任务。

### 2.5-A — 阻断性：扩展 NodeExecutor 接口

**问题**：`WorkflowNodeExecutor` 缺少 `Schema()` 与 `Resume()` 方法，`Execute` 返回值只有二态（成功/失败），无法表达 `manual-classifier` 所需的 `waiting_input` 挂起状态。

- [x] 定义 `ExecutionStatus` 三态常量：`ExecutionSuccess` / `ExecutionFailure` / `ExecutionPending`
- [x] `NodeExecutionOutput` 增加 `Status ExecutionStatus` 和 `PendingReason string` 字段
- [x] `WorkflowNodeExecutor` 接口新增 `Schema() NodeSchema` 方法（供 Phase 4 编辑器的 `GET /api/node-types` 端点使用）
- [x] `WorkflowNodeExecutor` 接口新增 `Resume(ctx context.Context, input NodeExecutionInput, resumeData map[string]any) (NodeExecutionOutput, error)` 方法（manual-classifier 用户确认后调用）
- [x] 现有三个 executor（`trigger` / `extRatioClassifier` / `move`）补充 `Schema()` 占位实现和 `Resume()` 返回 not-supported 错误，保持编译通过
- [x] `executeWorkflowRun` 识别 `ExecutionPending` 状态：将 WorkflowRun 更新为 `waiting_input`，停止当前轮循环；Resume API 触发后从 `resume_node_id` 处重入，并将用户数据传入 `Resume()` 方法

### 2.5-B — 阻断性：实现端口数据传播（Port-based Data Flow）

**问题**：当前执行器只接收 DB 中的 `Folder` 对象，节点间无数据传递通道。分类管道节点（`folder-tree-scanner` → `name-keyword-classifier` → `subtree-aggregator`）需要传递 `FolderTree` / `ClassificationSignal` 等类型，当前模型完全无法支持。

- [x] `WorkflowGraphNode` 结构体增加字段：`Label string`、`Inputs map[string]NodeInputSpec`、`UIPosition *NodeUIPosition`
- [x] `WorkflowGraphEdge` 结构体增加字段：`ID string`、`SourcePort int`、`TargetPort int`
- [x] 定义 `NodeInputSpec` 结构体：互斥字段 `ConstValue any` 或 `LinkSource *NodeLinkSource`（含 `SourceNodeID string` + `OutputPortIndex int`）
- [x] `NodeExecutionOutput.Output map[string]any` 改为 `Outputs []any`（按端口索引排列），兼容多输出端口节点（`confidence-check` high/low、`category-router` 五路）
- [x] `NodeExecutionInput` 新增 `Inputs map[string]any`（key = 端口名，value = 上游节点对应端口的输出值）
- [x] WorkflowRunner `executeWorkflowRun` 维护 `outputCache map[string][]any`（key = nodeID）：节点执行前按 `node.Inputs` 中的 `link_source` 从缓存拼装 `Inputs`，节点执行后将 `Outputs` 存入缓存
- [x] 现有三个 executor 的 `Execute` 返回值改为 `Outputs: []any{结果}`，内部实现可继续读 `input.Folder` 作为过渡，不强制立即迁移

### 2.5-C — 非阻断性：DB Schema 补齐（新增迁移 006）

**问题**：`005_workflow_engine.sql` 落地的表结构相比 `节点系统设计.md §八` 缺少若干字段，`manual-classifier`（Phase 3）和 `compress-node`（Phase 4）会用到。

- [x] 新建 `backend/internal/db/migrations/006_workflow_engine_ext.sql`，追加：
  - `workflow_definitions`：增加 `description TEXT NOT NULL DEFAULT ''`
  - `workflow_runs`：增加 `last_node_id TEXT`、`external_blocks INTEGER NOT NULL DEFAULT 0`、`error TEXT`、`started_at DATETIME`、`finished_at DATETIME`
  - `node_runs`：增加 `input_signature TEXT`（结果缓存 key）、`resume_token TEXT`（manual-classifier 恢复令牌）
  - `node_snapshots`：增加 `compensation TEXT`（compress-node `delete_source=true` 时的回退指令描述）；`kind` 新增合法值 `rollback_base`（仅文档约定，列类型不变）
- [x] 对应更新 `repository/models.go` 四个结构体，补充缺少字段
- [x] `WorkflowRunRepository` 补充 `UpdateBlocks(ctx, id, delta int) error`（`external_blocks` 增减，供 manual-classifier 使用）
- [x] `NodeRunRepository` 补充 `SetResumeToken(ctx, nodeRunID, token string) error`
